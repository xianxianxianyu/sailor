# Sailor 前端页面聚类方案

## 当前状态

现有 8 个独立页面：Sniffer、Trending、Tags、KB、Feeds、Follow、Boards、Research

用户期望：**4 个页面**（搜索、关注、知识库、订阅源）

## 聚类分析

根据文档设计意图，Sailor 是一个"信息供应链闭环"系统：
- 发现 → 订阅 → 知识库 → 追踪

8 个页面的功能关系：

```
Sniffer (主动搜索)  ──┐
Trending (趋势热点)  ──┤── 都是"发现内容"
                       │
Feeds (RSS订阅源)   ──┤── "持续输入"
                       │
KB (知识库)         ──┤── "存储管理"
Tags (标签系统)     ──┘
                       │
Follow (关注)       ──┤── "追踪变化"
Boards (看板)       ──┤
Research (研究计划)  ──┘
```

---

## 方案 A：按信息流阶段（推荐）

4 个页面：**发现 → 订阅源 → 知识库 → 追踪**

符合文档中"信息供应链闭环"的设计理念，每个页面对应一个阶段。

利弊分析：
- ✅ 搜索和趋势都是"发现新内容"，用户心智模型一致
- ✅ 趋势可以作为搜索的补充，用户不搜索时也能看到热点
- ✅ 两个Tab切换简单直观
- ⚠️ 搜索页本身已经比较复杂（搜索栏+结果+检查器），加上趋势后内容较多

│ │ ○ 论文源1 │ │                              │

│ └──────────┘ │ │ 文章1  文章2  文章3       │  │

- ✅ 左右分栏布局清晰，源列表+详情一目了然
- ✅ 与文档设计一致：订阅源是"持续输入"的核心
- ⚠️ 如果源很少，页面会显得空旷

## 方案 B：按用户角色

4 个页面：**搜索与浏览 → 关注与追踪 → 知识库 → 洞察**

按用户使用场景分：浏览者 → 追踪者 → 管理者 → 分析者

### 页面 1：搜索与浏览 (Search)

合并：Sniffer（搜索功能）

```
┌─────────────────────────────────────────────┐
│ ┌─搜索栏────────────────────────────────┐   │
│ │ 关键词 | 频道 | 时间 | 排序 | [搜索]  │   │
│ └───────────────────────────────────────┘   │
│                                             │
│ ┌─结果区域──────────┐ ┌─侧边栏──────────┐  │
│ │ 搜索结果卡片列表    │ │ 分析/摘要       │  │
│ │                     │ │ 趋势热点        │  │
│ │ □ 结果1 [分析]      │ │ (来自Trending)  │  │
│ │ □ 结果2 [分析]      │ │                  │  │
│ │ □ 结果3 [分析]      │ │ 热门标签1        │  │
│ │                     │ │  - 文章a         │  │
│ │                     │ │  - 文章b         │  │
│ │                     │ │ 热门标签2        │  │
│ │                     │ │  - 文章c         │  │
│ └─────────────────────┘ └──────────────────┘  │
│ [保存到KB] [转为订阅源] [对比分析]            │
└─────────────────────────────────────────────┘
```

利弊分析：
- ✅ 搜索是最高频操作，独占一页体验好
- ✅ 趋势热点放在侧边栏，不搜索时也有内容看
- ✅ 不需要Tab切换，信息密度高
- ⚠️ 侧边栏同时显示分析和趋势，可能信息过载
- ⚠️ 趋势内容在侧边栏空间有限，展示不够充分

### 页面 2：关注与追踪 (Follow)

合并：Follow + Boards + Research + Feeds

```
┌─────────────────────────────────────────────┐
│ ┌─左侧列表──────────┐ ┌─右侧详情────────┐  │
│ │ 分组:               │ │                  │  │
│ │ ▼ 订阅源            │ │ 选中项详情       │  │
│ │   RSS源1 ●          │ │                  │  │
│ │   RSS源2 ○          │ │ 订阅源: 配置+历史│  │
│ │   API源1 ●          │ │ 关注: 快照+对比  │  │
│ │ ▼ 关注              │ │ 看板: 排行+变化  │  │
│ │   关注1             │ │ 研究: 论文列表   │  │
│ │   关注2             │ │                  │  │
│ │ ▼ 看板              │ │ [手动运行]       │  │
│ │   GitHub趋势        │ │ [编辑] [删除]    │  │
│ │   HF论文            │ │                  │  │
│ │ ▼ 研究计划          │ │                  │  │
│ │   AI研究            │ │                  │  │
│ └─────────────────────┘ └──────────────────┘  │
│ [+添加订阅源] [+新建关注] [导入OPML]         │
└─────────────────────────────────────────────┘
```

利弊分析：
- ✅ 所有"输入源"统一管理，概念清晰
- ✅ 左侧分组折叠，可以容纳大量项目
- ✅ 右侧详情根据类型自适应显示
- ⚠️ 四种不同类型混在一起，详情面板需要根据类型切换
- ⚠️ 订阅源和关注的操作差异较大，合并可能增加复杂度
- ⚠️ 页面逻辑最复杂，开发工作量大

### 页面 3：知识库 (Knowledge Base)

合并：KB + Tags（与方案A相同）

│                                             │

利弊分析：（同方案A）
- ✅ 标签是KB的附属功能，弹窗管理合理
- ✅ 文章+图谱双视图覆盖所有需求

- ⚠️ 标签分析功能目前后端支持有限
- ⚠️ 与搜索页的趋势功能有重叠

## 方案 C：极简三页 + 全局侧边栏

3 个页面：**搜索 → 输入 → 管理**

最激进的精简方案，导航从顶部改为左侧图标栏。

### 全局布局

```
┌──────┬──────────────────────────────────────┐
│ 侧栏  │ 主内容区                             │
│      │                                      │
│ 🔍   │  (根据左侧选择切换)                   │
│ 搜索  │                                      │
│      │                                      │
│ 📡   │                                      │
│ 输入  │                                      │
│      │                                      │
│ 📚   │                                      │
│ 管理  │                                      │
│      │                                      │
│ ──── │                                      │
│ ⚙    │                                      │
│ 设置  │                                      │
└──────┴──────────────────────────────────────┘
```

### 页面 1：搜索 (Search)

只包含：Sniffer

```
┌──────────────────────────────────────────────┐
│ ┌─搜索栏──────────────────────────────────┐  │
│ │ 关键词 | 频道 | 时间 | [搜索]           │  │
│ └─────────────────────────────────────────┘  │
│ ┌─结果──────────────┐ ┌─面板──────────────┐  │
│ │ 结果卡片列表        │ │ 分析/摘要         │  │
│ │ [加入KB][订阅][对比] │ │                   │  │
│ └─────────────────────┘ └───────────────────┘  │
└──────────────────────────────────────────────┘
```

利弊分析：
- ✅ 搜索页最纯粹，只做搜索
- ✅ 没有Tab切换，操作最简单
- ⚠️ 趋势热点没有入口，需要用户主动搜索才能发现内容
- ⚠️ 对于不知道搜什么的用户，页面缺少引导

### 页面 2：输入 (Input)

合并：Feeds + Follow + Boards + Research

```
┌──────────────────────────────────────────────┐
│ [订阅源] [关注] [看板] [研究]  ← 子Tab       │
├──────────────┬───────────────────────────────┤
│ 列表          │ 详情                          │
│ (所有输入源)  │ (配置/历史/状态)               │
│              │                               │
│ + 添加       │ [运行] [编辑] [删除]           │
└──────────────┴───────────────────────────────┘
```

利弊分析：
- ✅ 所有"数据输入"统一管理
- ✅ 子Tab清晰分类
- ⚠️ 四种类型的操作差异大，统一界面有挑战
- ⚠️ 开发复杂度最高

### 页面 3：管理 (Manage)

合并：KB + Tags + Trending

```
┌──────────────────────────────────────────────┐
│ [知识库] [标签] [趋势报告]  ← 子Tab          │
├──────────────────────────────────────────────┤
│                                              │
│ 知识库Tab:                                    │
│  KB选择 → 文章/图谱                           │
│                                              │
│ 标签Tab:                                      │
│  标签CRUD + 权重管理                           │
│                                              │
│ 趋势报告Tab:                                  │
│  按标签分组的热门内容                           │
└──────────────────────────────────────────────┘
```

利弊分析：
- ✅ 页面数量最少（3页），导航最简洁
- ✅ 左侧图标栏节省空间
- ⚠️ "管理"页面承载太多功能，子Tab过多
- ⚠️ 趋势报告放在"管理"下不太直觉
- ⚠️ 三页可能不够，用户找功能需要多点一层

## 三种方案对比总结
